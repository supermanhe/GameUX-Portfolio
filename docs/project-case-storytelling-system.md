# 项目详情页叙事与呈现系统

## 当前诊断

首页与项目头部已经形成较成熟的暗色、高对比视觉语言，但案例内容仍主要依赖统一的“编号标题 + 三条亮点 + 长图/媒体连续平铺”。这会产生三个问题：

1. 所有案例节奏相同，无法体现付费优化、交互创新、规范搭建等工作的差异。
2. 设计结论藏在图片里，读者难以快速理解“为什么这样设计”。
3. 证据层级不清，截图、流程、方法和结果被当作同等重要的素材展示。

现有 13 个案例中，8 个正文几乎只有图片；`UEprocess` 单项包含 28 张图片。当前内容模型只有标题、亮点、媒体和正文，也不足以结构化表达角色、问题、决策、验证与结果。

## 新的内容原则

每个案例只讲清一个核心论点：

> 面向哪类玩家，在什么场景中，我通过什么关键判断改善了什么体验，并由什么证据支持。

每个主要设计点使用同一推理单元：

> 问题证据 -> 设计洞察 -> 关键决策 -> 验证证据 -> 结果或学习

统一的是论证质量，不是页面版式。不同案例应根据证据类型选择不同叙事原型。

## 五类案例原型

| 原型 | 适用案例 | 推荐叙事重点 |
|---|---|---|
| 转化优化型 | BP、付费货币不足、商城 | 摩擦漏斗、关键决策、指标结果 |
| 系统设计型 | 命卡、交易所、资源流通 | 玩家心智模型、信息架构、状态与规则 |
| 交互创新型 | 技能轮盘、社交玩法 | 场景约束、原型演示、可用性验证 |
| 设计基建型 | Figma 控件库、交互流程 | 组织问题、介入方法、覆盖范围、采用结果 |
| 探索迭代型 | 尚无强结果数据的概念 | 假设、方案取舍、测试学习、下一步 |

## 页面节奏

建议每个案例由 5 至 8 个独立网页区块组成，而不是一张完整长图：

1. **结果型开场**：一句核心论点、角色范围、关键证据、最强视觉。
2. **问题证据**：旧流程、玩家反馈、行为数据或高频摩擦。
3. **核心洞察**：说明真正改变方向的判断。
4. **决策聚焦**：只选 2 至 3 个决定，每个决定配一张聚焦图或短动效。
5. **验证与迭代**：展示测试任务、观察、失败版本与调整。
6. **结果**：数据必须标注周期、口径与限制；无数据时明确使用定性证据。
7. **反思**：保留的权衡与下一步假设。
8. **证据附录**：完整长图、流程稿和其他素材放入可展开画廊。

## 视觉模块

- `Outcome Hero`：大标题、成果数字、主视觉和角色范围。
- `Friction Map`：旧流程截图配编号标注，解释玩家在哪一步犹豫或流失。
- `Decision Spotlight`：左右交替排版，正文只解释一个决策。
- `Before / After`：仅用于能严格对齐的前后状态，避免为了形式硬对比。
- `Prototype Stage`：GIF 或视频优先，旁边写测试任务和观察。
- `Test Finding`：按问题严重度展示反馈，以及反馈如何改变方案。
- `Metric Card`：基线、结果、时间范围、样本/口径、限制。
- `Evidence Gallery`：完整长图留作二级证据，默认不打断主叙事。

## 长图处理方法

不要继续把信息、说明文字和截图全部画进一张图。将长图拆为：

1. 可被网页直接排版的标题、正文、数字和引语。
2. 能证明单个决策的截图裁切。
3. 需要轻量标注的局部对比图。
4. 适合动态表达的流程或反馈，改为 GIF/视频。
5. 完整原图放入“查看完整过程”或证据画廊。

网页应负责排版与叙述，图片只负责提供视觉证据。

## 数据与测试表达

优先级从高到低：

1. 上线前后指标、A/B 测试、分阶段发布结果。
2. 可用性测试任务完成率、耗时、错误率、SEQ/SUS 等。
3. 行为日志、客服反馈、社区反馈的重复主题。
4. 专家评审、团队采用率、规范覆盖率。
5. 明确标注为假设或后续验证计划的推断。

不要把项目总流水直接归因于单个 UX 决策。应写清个人贡献、指标口径与可能的混杂因素。

## 建议优先改造顺序

1. **BP 优化**：已有“付费率提升 10%+”，最适合建立数据驱动案例模板。
2. **技能轮盘迭代**：有费茨定律、横屏热区和量化规则，最适合展示交互独创性与可用性测试。
3. **从 0-1 构建交互流程**：素材最多但目前最像长图档案，适合改造成设计基建型案例。

先完成这三种原型，再将其模块复用到其他案例；不要一次性把 13 个案例套入同一新版模板。
